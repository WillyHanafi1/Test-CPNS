 Bagian ini sangat krusial karena berkaitan langsung dengan pendapatan websitemu (jika kamu memonetisasinya) dan keamanan konten agar soal-soal eksklusif tidak bocor ke pengguna yang tidak berhak.1. Backend Logic: API Endpoint & Middleware (Sang Satpam Server)Dalam arsitektur backend (misalnya jika kamu menggunakan FastAPI dengan Python), API endpoint bertindak sebagai jalur komunikasi, sedangkan Middleware adalah pos penjagaannya.API Endpoint (GET /api/packages): Ini adalah "buku menu". Ketika pengguna membuka halaman Katalog di website, browser mereka akan memanggil endpoint ini. Server kemudian akan merespons dengan format JSON yang berisi daftar lengkap paket ujian (nama, harga, deskripsi, gambar cover) untuk dirender oleh frontend menjadi tampilan visual yang menarik.Cara Kerja Middleware (Role-Based Access Control / RBAC): Misalkan ada paket bernama "Tryout SKD Premium". Pengguna harus membayar untuk bisa mengaksesnya. Apa yang terjadi jika pengguna gratisan (free tier) mencoba iseng mengubah URL di browser atau menembak API secara langsung menggunakan tools (misal: memanggil GET /api/exams/premium-1/start)?Intersepsi: Permintaan tersebut tidak akan langsung masuk ke fungsi utama untuk mengambil soal. Permintaan itu ditangkap lebih dulu oleh Middleware.Validasi Token: Middleware akan membaca token otorisasi (JWT) yang diselipkan di dalam Header permintaan.Pengecekan Izin: Middleware mengecek ke database atau memeriksa isi token tersebut: "Apakah pengguna dengan ID ini memiliki hak akses aktif untuk paket ID 'premium-1'?"Eksekusi 403 Forbidden: Jika sistem mendeteksi pengguna tersebut belum membayar, Middleware akan langsung memutus jalur dan mengembalikan kode status HTTP 403 Forbidden. Pesan error akan dikirim, misalnya: {"detail": "Akses ditolak. Silakan upgrade ke Premium untuk membuka paket ini."}. Frontend akan menangkap pesan ini dan menampilkan peringatan atau mengarahkan pengguna ke halaman pembayaran.2. Arsitektur Database: Relasi Tabel Paket dan TransaksiAgar Middleware bisa bekerja otomatis memisahkan mana pengguna berhak dan tidak, kita butuh struktur database relasional yang saling mengunci (menggunakan SQLAlchemy di PostgreSQL/MySQL).Berikut adalah rancangan skema tabelnya:Tabel exam_packages (Katalog Utama)Menyimpan semua daftar paket ujian yang kamu buat.id (Primary Key, misal: UUID)title (Contoh: "Tryout SKD Akbar BKN Batch 1")description (Penjelasan singkat isi paket)price (Angka integer, misal: 50000. Jika 0 berarti gratis)is_premium (Boolean: True / False)Tabel users (Data Pengguna)Menyimpan identitas akun peserta.id (Primary Key)nameemailTabel user_transactions (Tabel Jembatan / Akses Kontrol)Ini adalah tulang punggung sistem monetisasi dan keamananmu. Tabel ini menghubungkan pengguna dengan paket yang mereka pilih/beli.id (Primary Key)user_id (Foreign Key $\rightarrow$ terhubung ke tabel users)package_id (Foreign Key $\rightarrow$ terhubung ke tabel exam_packages)payment_status (Sangat penting! Gunakan tipe Enum: pending, success, failed, atau expired)access_expires_at (Batas kedaluwarsa akses, misalnya paket hanya bisa diakses selama 6 bulan sejak dibeli).Logika Penggabungan (The Magic):Ketika Middleware tadi melakukan "Pengecekan Izin", secara teknis SQLAlchemy di backend menjalankan perintah query logis seperti ini:"Cari di tabel user_transactions, apakah ada baris data di mana user_id adalah si Budi, package_id adalah Tryout Premium, DAN payment_status-nya adalah 'success', DAN waktu saat ini belum melewati access_expires_at?"Jika hasil query itu ditemukan (bernilai True), maka gembok dibuka dan soal dikirim ke browser Budi. Jika tidak ditemukan, satpam (Middleware) langsung melempar 403 Forbidden.
